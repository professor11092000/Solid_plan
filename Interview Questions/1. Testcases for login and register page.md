Here is comprehensive checklist for Registration page and login page.

## Registration Page — Test Cases

**Input Validation & Field-Level**
- Email field accepts valid formats only (reject malformed emails: missing @, no domain, etc.)
- Test for SQL injection, XSS, command injection in every field (name, email, username, etc.)
- Test special characters, Unicode, emoji in name/username fields — check for crashes or stored XSS
- Test extremely long input (buffer/field-length boundary testing) in every field
- Null byte injection, leading/trailing whitespace handling
- Password field enforces stated complexity policy (length, character classes) — test boundary cases (exactly minimum length, one character short, etc.)
- Confirm-password field actually validates match server-side, not just client-side JS

**Business Logic / Account Creation**
- Duplicate email/username registration — is it blocked server-side, not just client-side?
- Case sensitivity handling on email (Test@x.com vs test@x.com should usually be treated as the same account)
- Email/phone verification bypass — can you access the account or its features before verifying?
- Race condition — submit the same registration request multiple times simultaneously (can you create duplicate accounts or bypass uniqueness checks?)
- Mass account creation / lack of CAPTCHA or rate limiting (bot registration abuse)
- Disposable/temp-mail domains accepted (policy-dependent, but worth flagging)
- Can you register with another real user's email and cause account confusion or denial-of-service for them?
- Privilege/role assignment — can a client-side parameter (e.g., `role=admin` in the request) escalate the account's role at signup? (Mass assignment vulnerability)
- IDOR on email verification link/token — predictable or guessable verification tokens
- Verification token reuse — does the same token work multiple times, or for a different account?
- Verification token expiry — does it ever expire, and is expired-token handling correct?

**Password Handling**
- Password strength meter (if present) matches actual server-side enforcement
- Common/breached password rejection (e.g., against a known-bad password list)
- Password is not reflected back in any response (API response, email, error message)
- Password not logged in server logs, analytics, or error tracking tools

**Transport & Storage**
- Registration request sent over HTTPS only (no HTTP fallback)
- Response doesn't leak whether an email is "already registered" in a way that enables **user enumeration** (a very common finding — error messages like "email already exists" let an attacker map valid accounts)

## Login Page — Test Cases

**Authentication Logic**
- SQL injection / NoSQL injection in username and password fields (classic `' OR '1'='1` style payloads, and blind/time-based variants)
- Authentication bypass via parameter manipulation (e.g., modifying a hidden field, manipulating a JSON body like `{"isAdmin":true}`)
- Test default/common credentials if applicable (admin/admin, etc.)
- Case sensitivity of username/email at login
- Login with unverified account — should usually be blocked until verification, test if it isn't
- Login with disabled/deleted/locked account — does it correctly reject?

**Brute Force & Rate Limiting**
- Account lockout policy after N failed attempts — does it trigger, and does it reset correctly?
- Rate limiting on login endpoint (test via Burp Intruder or similar) — IP-based and account-based
- CAPTCHA bypass (is it actually enforced server-side, or just a client-side widget that can be skipped by calling the API directly?)
- Username enumeration via different error messages or response times for "user not found" vs. "wrong password"
- Username enumeration via response time difference (timing attack) even when error messages are identical

**Session Management (critical area)**
- Session token is unpredictable, sufficiently random, and not sequential
- Session token issued only after successful authentication — not before
- Old session invalidated on new login (no session fixation)
- Logout actually invalidates the session server-side (test by reusing the token after logout — it should fail)
- Concurrent session handling — does logging in from a second device invalidate the first, or is that even checked?
- Session timeout — idle and absolute timeout enforced correctly
- Session token not exposed in URL (GET parameters), browser history, or logs
- Cookie flags set correctly: `HttpOnly`, `Secure`, `SameSite`
- JWT-specific (if used): signature verification enforced, `alg:none` rejected, expiry (`exp`) enforced, no sensitive data in unencrypted payload

**MFA / 2FA (if present)**
- MFA can't be bypassed by directly calling the post-login endpoint, skipping the OTP step
- OTP brute-forceable (4-6 digit codes without rate limiting)
- OTP reusable after successful use, or after expiry
- "Remember this device" logic doesn't become a permanent bypass

**Password Reset / Forgot Password (closely tied to login)**
- Reset token is sufficiently random, not predictable/sequential
- Reset token expires in a reasonable time window
- Reset token invalidated after first use
- Reset token tied to the specific account it was issued for (test using it to reset a *different* account)
- Host header injection in password reset email links (can attacker control the reset link domain via a manipulated Host header?)
- Old password/session invalidated after a successful reset

## Cross-Cutting / Both Pages

- CSRF protection on both forms (if state-changing and cookie-based auth is used)
- Clickjacking protection (`X-Frame-Options` / CSP `frame-ancestors`) on login/registration pages
- Autocomplete settings on password fields (policy-dependent — usually should allow password managers, but disable for shared/kiosk-mode use cases)
- Verbose error messages leaking stack traces, server info, or internal details
- Response headers checked: no sensitive data in headers, proper security headers present (CSP, HSTS, X-Content-Type-Options)
- Client-side validation can be bypassed entirely and server still enforces the same rules (the golden rule: never trust the client)
- Check for sensitive data exposure in browser dev tools / network tab during the flow (tokens, internal IDs, etc.)
- Open redirect on post-login/post-registration redirect parameter (`?redirect=`/`?next=`)

### What special you can bring it on the table, this all testcases follow by every pentester.

Most pentesters run that list as independent checks: "is rate limiting present? yes/no. Is the token random? yes/no." That's necessary but it's checklist-following, not threat modeling. The gap is in three places:
1. You're not chaining findings into an actual attack
Individually-low-severity findings on a login page combine into critical ones. A "special" approach is to explicitly hunt for chains, not just tick boxes:

**User enumeration (low) + no rate limiting (medium) + weak password policy (low) = practical account takeover (critical)**. Most testers report these as three separate low/medium findings. The chain is the real risk, and almost nobody writes that up explicitly with a PoC that walks start-to-finish.
Verbose registration error ("email already exists") + password reset flow → you don't even need to brute-force; you enumerate first, then go straight to reset.   
Test this by actually building the chain end-to-end and reporting it as one critical issue with a kill-chain narrative, not three checklist rows.   
   
2. Almost nobody actually tests state/race conditions properly   
"Race condition" is on every checklist as a bullet point, but most testers never actually fire concurrent requests — they mention it, don't test it, and move on. Doing it properly is where you differentiate:   
  
Use Burp's Turbo Intruder or race the web (single-packet attack technique by James Kettle) to send genuinely simultaneous requests, not just "fast sequential" ones via normal Intruder — sequential isn't a real race condition test.  
Apply this specifically to: registration (duplicate accounts/free-credit abuse), password reset (token generation race), MFA verification (single OTP used twice), and account lockout counters (can you out-pace the lockout by racing requests before the counter increments?).
This single technique catches bugs that 90% of "I checked the box" pentests miss entirely.  
  
3. You're testing the login form, not the login system  
**Real depth comes from going one layer below the form:**  

Differential testing across channels — if there's a mobile app, a web app, and maybe a partner API all hitting the same auth backend, test whether the weakest one's validation logic is actually the one enforced everywhere. A mobile API often has laxer checks than the web flow for the exact same account creation endpoint — you already know this from our mobile conversation, and that crossover insight is exactly the differentiator you're equipped to bring.
Out-of-band side channels: does the registration confirmation email or SMS leak the verification token in a way a different vulnerability (e.g., an open redirect, or a third-party analytics SDK capturing the URL) could exfiltrate?
Password reset + OAuth/SSO interaction: if the app supports "Login with Google" alongside email/password, test whether resetting the password-based account can be used to hijack an account that was originally created via SSO with the same email — a classic account-linking confusion bug that almost never shows up on a generic checklist.

4. Logic abuse that requires actually understanding the business, not just the form   
  
If registration grants a signup bonus, referral credit, or free trial — that's a business-logic target, not just an input-validation target. Test referral-code reuse, self-referral, and multi-account farming.  
If login throttling is IP-based only, test bypass via header spoofing (X-Forwarded-For) — extremely common gap, rarely verified end-to-end with an actual distributed-looking request set.  


The honest framing: the checklist gets you to "competent," not "differentiated." What separates a thorough tester from an average one on this exact page isn't a longer list of test cases — it's (a) chaining low findings into a real attack path with a PoC, (b) actually executing race-condition tests instead of just mentioning them, and (c) bringing cross-system thinking (mobile vs. web vs. SSO) into a page that looks like "just a form" to someone testing it in isolation.
