## 1. What is Threat Modeling

**Threat modeling** is a structured, proactive process used to identify, communicate, and understand threats and mitigations within the context of protecting an asset (an application, system, or business process).

Instead of asking *"what vulnerabilities exist?"* (the SAST/DAST question), threat modeling asks:

> "What can go wrong with this system, and what are we doing about it?"

It is done **early** in the design phase (ideally before a single line of code is written), although it is also valuable for existing systems.

**Key idea:** Threat modeling is about *thinking like an attacker, structurally*, not randomly guessing. It produces a list of threats mapped to mitigations — which then often *becomes input* for your SAST rules, DAST test cases, and pen test scope.

| Aspect | Threat Modeling | SAST | DAST | Pen Testing |
|---|---|---|---|---|
| When | Design phase (ideally) | Code commit/build | Pre-prod/Staging | Pre-release/Periodic |
| What it finds | Design flaws, missing controls | Code-level bugs (insecure code patterns) | Runtime vulnerabilities | Exploitable real-world issues |
| Output | Threats + mitigations list | List of code findings | List of runtime findings | Exploit proof + report |
| Cost to fix | Cheapest (design stage) | Cheap | Moderate | Expensive |

---

## 2. Why Threat Modeling Matters in AppSec

- **Shifts security left** — design flaws caught before code is written are exponentially cheaper to fix than bugs caught in production.
- **SAST/DAST can't find everything** — they find implementation bugs (e.g., SQLi, XSS). They **cannot** find:
  - Missing authorization checks by design (e.g., no concept of "owner" enforced at API layer)
  - Business logic flaws
  - Trust boundary violations (e.g., trusting a JWT claim without re-verifying server-side)
  - Architectural weaknesses (e.g., a single compromised microservice can pivot to all others)
- **Compliance** — required/recommended under PCI-DSS, NIST SSDF, ISO 27001, OWASP SAMM, Microsoft SDL.
- **Prioritization** — helps security teams focus pen testing and SAST/DAST scope on the highest-risk components instead of treating the whole app equally.

## 3. Core Terminology

| Term | Definition |
|---|---|
| **Asset** | Anything of value to be protected — data, credentials, system, reputation. |
| **Threat** | A potential event/actor that could cause harm to an asset (e.g., "attacker spoofs identity"). |
| **Threat Actor** | The entity carrying out the threat (script kiddie, insider, nation-state, competitor). |
| **Vulnerability** | A weakness that can be exploited by a threat to cause harm. |
| **Risk** | The likelihood × impact of a threat exploiting a vulnerability. |
| **Attack Vector** | The path/means by which an attacker gains access (e.g., phishing email, exposed API endpoint). |
| **Attack Surface** | The sum of all points where an attacker could try to enter or extract data. |
| **Trust Boundary** | A point where data/control crosses from one trust level to another (e.g., internet → DMZ, user → server). |
| **Data Flow** | Movement of data between processes, stores, and external entities. |
| **Mitigation/Control** | A safeguard reducing likelihood or impact of a threat. |
| **DFD (Data Flow Diagram)** | Visual representation of how data moves through a system — the primary input artifact for most threat modeling methods. |
| **Entry Point** | Any place where external input enters the system. |
| **Exit Point** | Any place where the system sends data out. |
| **Security Control** | Preventive, detective, or corrective measure (e.g., WAF, MFA, logging). |
| **Residual Risk** | Risk remaining after mitigations are applied. |
| **Attack Tree** | Hierarchical diagram showing how a goal (root) can be achieved via various attack paths (leaves). |
| **Kill Chain** | Sequential stages an attacker goes through (recon → weaponize → deliver → exploit → C2 → exfiltrate). |
| **Abuse Case** | The "evil twin" of a use case — describes how a feature can be misused. |
| **Defense in Depth** | Layered security controls so no single failure leads to compromise. |
| **Least Privilege** | Entities should have only the minimum access necessary. |
| **Blast Radius** | The scope of damage if a particular component is compromised. |
| **Crown Jewels** | The most critical/high-value assets in a system (e.g., customer PII DB, signing keys). |

## 4. The Threat Modeling Process (4 Question Framework)

Adam Shostack's widely used four-question framework (used by Microsoft, OWASP):

1. **What are we building?**
   → Build a Data Flow Diagram / Architecture diagram. Identify components, data stores, external entities, trust boundaries.

2. **What can go wrong?**
   → Apply a methodology (commonly STRIDE) to systematically identify threats per component/data flow.

3. **What are we going to do about it?**
   → Define mitigations: accept, mitigate, transfer, or avoid the risk.

4. **Did we do a good job?**
   → Validate the model — review with stakeholders, re-test after implementation, update model as system evolves.

This is **cyclical**, not one-time — re-run threat modeling whenever architecture changes significantly.

## 5. Threat Modeling Methodologies

### 5.1 STRIDE (Microsoft) — Most commonly asked about in interviews

STRIDE categorizes threats into 6 types. Apply it per **element** in the DFD (process, data store, data flow, external entity).

| Letter | Threat | Violates | Example |
|---|---|---|---|
| **S** | Spoofing | Authentication | Attacker pretends to be another user using a stolen JWT |
| **T** | Tampering | Integrity | Attacker modifies a price parameter in an API request |
| **R** | Repudiation | Non-repudiation | User denies performing a transaction due to no audit log |
| **I** | Information Disclosure | Confidentiality | API leaks PII in error messages or verbose stack traces |
| **D** | Denial of Service | Availability | Attacker floods login endpoint to exhaust resources |
| **E** | Elevation of Privilege | Authorization | Regular user accesses admin endpoint due to missing role check (BOLA/IDOR) |

**Mnemonic:** *"Some Threats Require Immediate Detection & Elimination."*

> **Interview tip:** STRIDE maps almost directly to security properties: Authentication, Integrity, Non-repudiation, Confidentiality, Availability, Authorization (AINCAA).

---

### 5.2 DREAD (Risk Rating Model)

Used to **rank/prioritize** threats found (often after STRIDE). Each threat is scored 1–10 (or High/Med/Low) on 5 factors:

| Factor | Question |
|---|---|
| **D**amage | How bad is the impact if exploited? |
| **R**eproducibility | How easily can it be reproduced? |
| **E**xploitability | How much effort/skill is needed to exploit? |
| **A**ffected Users | How many users/systems affected? |
| **D**iscoverability | How easily can attacker find this flaw? |

**Risk Score = (D+R+E+A+D) / 5**

> Note: DREAD is criticized for subjectivity — many teams now prefer **CVSS** or **risk matrices** (Likelihood × Impact) instead, but it still comes up frequently in interviews as a classic model.

---

### 5.3 PASTA (Process for Attack Simulation and Threat Analysis)

A **risk-centric**, 7-stage methodology that ties business objectives to technical threats — popular in mature/enterprise AppSec programs.

1. Define Objectives (business goals, compliance needs)
2. Define Technical Scope (architecture, components, dependencies)
3. Application Decomposition (DFDs, trust boundaries, use cases)
4. Threat Analysis (threat intelligence correlation)
5. Vulnerability & Weakness Analysis (map to CVEs/CWEs — **this is where SAST/SCA output feeds in directly**)
6. Attack Modeling (attack trees/attack libraries, simulate attacker paths)
7. Risk & Impact Analysis (business impact, residual risk, countermeasures)

> **Interview tip:** PASTA is attacker-centric AND business-risk-centric — it's heavier than STRIDE but gives a risk score tied to business impact, which executives like.

---

### 5.4 VAST (Visual, Agile, Simple Threat Modeling)

Designed for **scalability** in Agile/DevOps environments. Splits into:
- **Application Threat Models** — based on process flow diagrams, for developers.
- **Operational Threat Models** — based on architecture, attacker perspective, for infra/ops teams.

Goal: integrate threat modeling into sprints without slowing down CI/CD (commonly cited when asked "how do you threat model in Agile/DevSecOps?").

---

### 5.5 Attack Trees

A **graphical method**: root node = attacker's goal (e.g., "Compromise admin account"), child nodes = ways to achieve it (AND/OR logic), leaves = specific attack steps.

```
Goal: Compromise Admin Account
├── OR: Phishing admin credentials
├── OR: Brute-force weak password
├── OR: Exploit session fixation
└── AND: Steal JWT token
     ├── Intercept via XSS
     └── Decode unsigned/weak JWT (alg=none)
```

Useful for visualizing **multiple attack paths** to a single high-value goal — often combined with STRIDE.

---

### 5.6 Trike

A **risk-based, requirements-driven** methodology focused on defining acceptable risk per stakeholder, then generating threats from a "requirements model" — less common in interviews, but good to mention as breadth.

---

### 5.7 OCTAVE (Operationally Critical Threat, Asset, and Vulnerability Evaluation)

Organization-wide, asset-driven risk assessment (CERT/Carnegie Mellon) — more aligned with **enterprise risk management** than a single application; focuses on organizational assets and operational risk rather than a specific app's DFD.

---

### 5.8 LINDDUN (Privacy Threat Modeling)

The privacy-focused counterpart to STRIDE — important if your firm deals with GDPR/PII.

| Letter | Threat |
|---|---|
| **L** | Linkability |
| **I** | Identifiability |
| **N** | Non-repudiation (in privacy context — user wants repudiation/deniability) |
| **D** | Detectability |
| **D** | Disclosure of information |
| **U** | Unawareness |
| **N** | Non-compliance |

> **Interview tip:** If asked "how would you threat model a system handling personal health data," mentioning LINDDUN alongside STRIDE shows depth.

## 6. Data Flow Diagrams (DFDs)

DFDs are the foundational diagram type for STRIDE-based threat modeling. Four basic elements:

| Symbol | Element | Meaning |
|---|---|---|
| Circle/Rounded box | **Process** | Code/component that transforms data (e.g., API service) |
| Two parallel lines | **Data Store** | Database, file, cache, queue |
| Rectangle | **External Entity** | User, third-party system, external API |
| Arrow | **Data Flow** | Movement of data between elements |
| Dashed line | **Trust Boundary** | Where trust level changes |

**Levels of DFD:**
- **Level 0 (Context Diagram)** — entire system as one process, external entities only.
- **Level 1** — breaks system into major components/processes.
- **Level 2+** — drills into sub-processes (used for complex systems).

> **Practical tip from your role:** A DFD is essentially the architecture diagram you'd want before doing a pen test scoping call — it tells you exactly where the entry points and trust boundaries are, which is exactly what you map your test cases to.

## 7. Trust Boundaries

A trust boundary is any point where data crosses between two zones of differing trust levels. **This is the single most important threat-modeling concept for interviews** because most real-world vulnerabilities (IDOR, SSRF, privilege escalation) occur at trust boundary violations.

Common trust boundaries in modern apps:
- Internet ↔ Load Balancer / WAF
- Frontend (browser) ↔ Backend API
- API Gateway ↔ Microservice
- Microservice ↔ Microservice (different teams/trust levels)
- Application ↔ Database
- On-prem ↔ Cloud
- Your org ↔ Third-party API/vendor
- Container ↔ Host
- User role boundary (regular user ↔ admin)

> **Golden rule:** "Never trust input that crosses a trust boundary without re-validating it on the more-trusted side." This is why client-side validation alone is insufficient — the browser-to-server line *is* a trust boundary.

## 8. Threat Modeling for APIs & Microservices

Since this is directly relevant to your API security work:

### Key API-specific threats (mapped to OWASP API Security Top 10):
| STRIDE Category | OWASP API Top 10 (2023) Mapping |
|---|---|
| Spoofing | API2: Broken Authentication |
| Tampering | API3: Broken Object Property Level Authorization |
| Repudiation | Lack of logging on API gateway |
| Info Disclosure | API3, API8: Security Misconfiguration (excessive data exposure) |
| DoS | API4: Unrestricted Resource Consumption |
| Elevation of Privilege | API1: BOLA (Broken Object Level Authorization), API5: BFLA (Broken Function Level Authorization) |

### Microservices-specific considerations:
- **East-west traffic** (service-to-service) often lacks the scrutiny given to north-south (internet-facing) traffic — a major blind spot.
- **Service mesh** (Istio, Linkerd) — mTLS between services is a mitigation for spoofing/tampering at internal trust boundaries.
- **API Gateway** as a chokepoint — centralizing auth here reduces per-service duplication of security logic, but also becomes a single point of failure if misconfigured.
- **JWT-specific threats**: algorithm confusion (alg=none), weak signing secrets, missing expiry validation, sensitive data in unencrypted JWT payload.
- **SSRF via webhooks/callbacks** is a very common modern API threat that's purely a design issue — exactly the kind of thing STRIDE/PASTA catches before code is written, but SAST often misses.

## 9. Threat Modeling for Cloud & DevSecOps (Shift Left)

Since you work with Checkmarx (SAST/SCA/IaC), this ties directly in:

- **IaC threat modeling**: review Terraform/CloudFormation templates for misconfigurations *before* deployment (e.g., overly permissive IAM roles, public S3 buckets, missing encryption) — this is "shifting threat modeling left" into the IaC scan stage.
- **Threat modeling as code**: tools like **pytm** and **Threagile** let you define your architecture in code/YAML and auto-generate STRIDE-based threat reports — useful for CI/CD integration.
- **Continuous Threat Modeling**: instead of a one-time exercise, teams maintain a "living" threat model updated via PRs whenever architecture changes (popularized by companies like Autodesk's "Continuous Threat Modeling" methodology).
- **Cloud-specific trust boundaries**: account boundaries (multi-account AWS/Azure), VPC boundaries, IAM role assumption chains, shared responsibility model (cloud provider vs. customer).

## 11. How Threat Modeling Connects to SAST/DAST/SCA

Since this is your exact world, here's the practical linkage to make in interviews:

| Threat Modeling Output | Feeds Into |
|---|---|
| Identified entry points / attack surface | **DAST scope** — which endpoints to crawl/test |
| Identified trust boundaries with missing validation | **SAST custom rules** — e.g., flag any input crossing a boundary without sanitization |
| Third-party/dependency risk (data store/external entity) | **SCA** — prioritize which vulnerable dependencies are actually reachable/exploitable based on the data flow |
| IaC misconfigurations identified as threats (e.g., public storage) | **IaC scanning rules** in Checkmarx |
| High-risk components (crown jewels) | **Pen test prioritization** — focus manual testing effort here |
| Missing authorization at a boundary | **Pen test test case**: explicitly test for IDOR/BOLA at that boundary |

> **This is a great line to use in interviews**: *"Threat modeling tells us where to point our SAST/DAST/pen-testing effort and why — it turns scanning from 'scan everything equally' into 'scan with context of what matters and what's reachable.'"*

## 12. Mitigations Mapping (STRIDE → Controls)

| Threat | Common Mitigations |
|---|---|
| **Spoofing** | MFA, strong authentication (OAuth2/OIDC), mutual TLS, certificate pinning, session management best practices |
| **Tampering** | Input validation, integrity checks (HMAC/checksums), digital signatures, parameterized queries, code signing |
| **Repudiation** | Centralized audit logging, tamper-evident logs (write-once), timestamps, digital signatures on transactions |
| **Information Disclosure** | Encryption at rest/in transit (TLS 1.2+), proper error handling (no stack traces to client), data minimization, access controls, masking/tokenization of PII |
| **Denial of Service** | Rate limiting, throttling, WAF, auto-scaling, CDN, resource quotas, circuit breakers |
| **Elevation of Privilege** | Least privilege, RBAC/ABAC, proper authorization checks at every layer (not just UI), input validation, sandboxing |

## 13. MITRE ATT&CK and Threat Modeling

While STRIDE/PASTA are **design-time, proactive** frameworks, **MITRE ATT&CK** is a **post-compromise, knowledge-base** of real-world adversary Tactics, Techniques, and Procedures (TTPs) observed in the wild.

| | Threat Modeling (STRIDE/PASTA) | MITRE ATT&CK |
|---|---|---|
| Purpose | Design-time risk identification | Real-world attacker behavior catalog |
| Used by | AppSec/architects during design | SOC/Red teams/Detection engineering |
| Granularity | Category-level (e.g., "Tampering") | Specific technique (e.g., T1190 "Exploit Public-Facing Application") |

**Interview tip:** A mature program **combines both** — use STRIDE to identify *what* threat categories apply to your architecture, then use ATT&CK to validate *how* real attackers have actually executed those threats, and to inform red team scenarios.

## 14. Common Mistakes in Threat Modeling

1. **Doing it once and never updating** — architecture changes, threat model should too (living document).
2. **Boiling the ocean** — trying to model every single threat for every component instead of prioritizing by risk/crown jewels.
3. **Treating it as a checklist exercise** rather than collaborative discussion with developers/architects.
4. **Skipping the DFD** and jumping straight to a list of generic threats — without a diagram, threats aren't grounded in the actual system.
5. **No follow-through** — identifying threats but not tracking mitigations to closure (no Jira ticket, no re-test).
6. **Ignoring trust boundaries within the system** (assuming internal traffic is safe).
7. **Doing it only at the application layer**, ignoring infrastructure/cloud/IaC threats.
8. **Confusing it with a vulnerability assessment** — threat modeling is about design-level "what could go wrong," not finding existing bugs.

## 15. Sample Worked Example (E-commerce API)

**System**: Customer-facing REST API → API Gateway → Order Service → Payment Service (third-party) → Order DB

**Step 1 — DFD elements:**
- External Entity: Customer (browser/mobile app)
- Process: API Gateway, Order Service, Payment Service Integration
- Data Store: Order DB
- Trust Boundaries: Internet↔Gateway, Gateway↔Order Service, Order Service↔Payment Service (third-party)

**Step 2 — STRIDE per element (sample):**

| Element | Threat | STRIDE | Mitigation |
|---|---|---|---|
| API Gateway | Attacker replays stolen JWT after logout | Spoofing | Short token TTL + token revocation list |
| Order Service | User modifies `order_total` in request body | Tampering | Server recalculates price server-side, never trusts client-supplied price |
| Order Service → Payment Service | No logging of who initiated refund | Repudiation | Centralized audit log with user ID + timestamp on all financial actions |
| Order DB | Verbose SQL error returned to client reveals schema | Info Disclosure | Generic error messages, detailed logs server-side only |
| API Gateway | Attacker floods `/checkout` endpoint | DoS | Rate limiting per IP/user, WAF rules |
| Order Service | Regular user accesses `/admin/orders/all` due to missing role check | Elevation of Privilege | Enforce RBAC checks server-side on every endpoint, not just UI hiding |

**Step 3 — Prioritize (DREAD or risk matrix)** → Elevation of Privilege scores highest (high impact + high discoverability) → becomes top pen-test/SAST custom-rule priority.

## 16. Interview Questions & Answers — Basic

**Q1: What is threat modeling?**
A: A structured process to identify, prioritize, and mitigate potential security threats to a system, typically performed during the design phase by analyzing how data flows through the system and where it could be attacked.

**Q2: What is the difference between a threat, a vulnerability, and a risk?**
A: A **vulnerability** is a weakness (e.g., missing input validation). A **threat** is a potential event that could exploit that weakness (e.g., an attacker injecting SQL). **Risk** is the likelihood of that threat occurring multiplied by its impact.

**Q3: What does STRIDE stand for?**
A: Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege — a Microsoft-developed taxonomy for classifying threats.

**Q4: Why is threat modeling done early in the SDLC?**
A: Because fixing a design flaw after deployment is far more expensive than fixing it on a whiteboard — "shift left" principle. It also prevents structural issues (like missing authorization layers) that no amount of later code scanning can fully catch.

**Q5: What is a Data Flow Diagram and why is it used in threat modeling?**
A: A DFD visually represents how data moves between external entities, processes, and data stores. It's used because most threat modeling methodologies (notably STRIDE) require systematically walking through each diagram element to ask "what can go wrong here."

**Q6: What's a trust boundary? Give an example.**
A: A point where data crosses between differing levels of trust — e.g., the line between a user's browser and your backend server. Any data crossing this boundary must be re-validated server-side, since client-side controls can be bypassed.

**Q7: Is threat modeling a one-time activity?**
A: No — it should be a living process, revisited whenever the architecture changes significantly (new feature, new integration, new trust boundary).

**Q8: What's the difference between threat modeling and penetration testing?**
A: Threat modeling is a proactive, design-time exercise that identifies *potential* threats and mitigations on paper. Penetration testing is a reactive, post-build exercise that attempts to *actually exploit* a running system to confirm real vulnerabilities exist.

## 17. Interview Questions & Answers — Intermediate

**Q9: Walk me through how you'd threat model a new feature.**
A: 
1. Understand the feature and draw/update the DFD (entities, processes, data stores, trust boundaries).
2. Apply STRIDE to each element, asking what could go wrong (e.g., for a data store: could it be tampered with, could it leak info?).
3. List threats, then rate them (DREAD or simple High/Med/Low risk matrix).
4. For each, define mitigation: fix in design, accept the risk, transfer it (e.g., via a WAF), or avoid the feature path entirely.
5. Document and track mitigations through to implementation and verification (ideally tied to SAST rules or test cases).
6. Review with stakeholders (architects, developers, security team).

**Q10: How does DREAD scoring work, and what are its drawbacks?**
A: Score 1-10 across Damage, Reproducibility, Exploitability, Affected users, Discoverability, then average for a risk score. Drawback: highly subjective — two people can score the same threat very differently, so many orgs now prefer CVSS-style or simple Likelihood × Impact matrices for consistency.

**Q11: What's the difference between STRIDE and PASTA?**
A: STRIDE is a **threat categorization** technique — quick, technical, per-component. PASTA is a full **7-stage risk-centric methodology** that ties technical threats back to business impact and uses real threat intelligence/attack simulation — heavier weight, used in mature/enterprise programs where business risk reporting matters.

**Q12: How would you threat model a microservices architecture differently from a monolith?**
A: In a monolith, the main trust boundary is typically just client↔server. In microservices, you must also threat model **east-west (service-to-service) traffic** — each service boundary is a potential trust boundary, especially if services have different trust levels or are owned by different teams. Service mesh / mTLS, centralized API gateway authentication, and per-service authorization checks become critical, since a single compromised service shouldn't be able to freely call any other internal service.

**Q13: How does threat modeling complement SAST and DAST?**
A: SAST/DAST find *implementation* bugs in existing code/running apps. Threat modeling finds *design* flaws — like a missing authorization layer or an untrusted trust boundary — *before* code exists, and it also tells you **where** to focus your SAST/DAST/pen-test effort by identifying the highest-risk components and entry points, since those tools can't tell you which findings matter most from a business-risk perspective on their own.

**Q14: What is an attack tree and when would you use one?**
A: A hierarchical diagram with the attacker's ultimate goal as the root and various attack paths as branches (using AND/OR logic). Useful when you want to visualize *all the ways* an attacker could reach a specific high-value goal (e.g., "compromise admin account"), which is harder to see using STRIDE's component-by-component approach alone.

**Q15: What is LINDDUN and when would you use it instead of/alongside STRIDE?**
A: LINDDUN is a privacy-focused threat modeling methodology (Linkability, Identifiability, Non-repudiation, Detectability, Disclosure, Unawareness, Non-compliance). You'd use it alongside STRIDE when the system handles significant personal/sensitive data (health records, biometric data) and you need to address privacy threats (e.g., GDPR) in addition to traditional security threats.

**Q16: How do you decide which mitigations to implement vs. accept the risk?**
A: Based on the risk rating (impact × likelihood) weighed against the cost/feasibility of mitigation, business context, and compliance requirements. Low-risk, high-cost-to-fix items might be formally "risk accepted" by a business owner with sign-off; high-risk items get prioritized for immediate mitigation regardless of cost.

**Q17: What's a common blind spot when threat modeling APIs specifically?**
A: Assuming the front-end's authorization/UI restrictions are sufficient — many teams hide an admin button in the UI but forget the underlying API endpoint still has no server-side role check, leading to BOLA/BFLA (Broken Object/Function Level Authorization). Threat modeling explicitly forces you to verify authorization is enforced at every trust boundary the request crosses, not just where the UI shows/hides controls.

## 18. Interview Questions & Answers — Advanced / Scenario-Based

**Q18: You're asked to threat model a system using JWT-based authentication across multiple microservices behind an API Gateway. What threats would you specifically look for?**
A:
- **Spoofing**: weak/no signature verification, algorithm confusion attacks (`alg: none`), leaked signing secret.
- **Tampering**: services trusting JWT claims (like `role` or `user_id`) without re-validating against the source of truth, especially if a downstream service can be reached directly bypassing the gateway.
- **Information Disclosure**: sensitive data placed unencrypted in the JWT payload (JWTs are base64-encoded, not encrypted).
- **Elevation of Privilege**: missing expiry/revocation checks allowing privilege to persist after role downgrade or logout; insufficient scope/audience validation allowing a token issued for Service A to be replayed against Service B.
- **Mitigation focus**: short-lived tokens + refresh token rotation, strict algorithm allow-listing, signature verification at every service (not just the gateway) if zero-trust internal architecture, audience (`aud`) claim validation per service.

**Q19: How would you threat model a system you didn't design and have very limited documentation for (common in legacy/brownfield apps)?**
A: Start with **architecture discovery**: review network diagrams, talk to developers/ops, or reverse-engineer the DFD from code/traffic analysis (e.g., reviewing API gateway logs, network traffic, IaC templates). Build at least a Level-0/Level-1 DFD even if imperfect. Prioritize based on **crown jewels** (where is sensitive data stored/processed) rather than trying to model the entire legacy estate at once. Use existing SAST/DAST/SCA findings as a starting signal for where weaknesses might cluster, then validate trust boundaries around those areas first.

**Q20: How do you scale threat modeling across hundreds of microservices/teams without becoming a bottleneck (Agile/DevSecOps context)?**
A:
- Adopt **lightweight, repeatable templates** per service type rather than a full bespoke STRIDE exercise every time (similar to VAST's philosophy).
- Use **"threat modeling as code"** tools (pytm, Threagile) so models live in version control alongside architecture-as-code and get regenerated automatically on PRs that touch architecture.
- Train developers to self-serve basic threat modeling for low-risk services; reserve security team deep-dives for crown-jewel/high-risk services.
- Maintain a **shared threat library** (common threats + mitigations per component type — e.g., "every Postgres data store gets these 5 standard threats by default") to avoid reinventing analysis each time.
- Integrate findings into existing backlogs (Jira) and gate releases on critical/high findings only, to avoid blocking velocity for low-risk items.

**Q21: A developer says "we already do SAST and DAST, why do we need threat modeling too?" How do you respond?**
A: SAST and DAST find *known patterns of bad code or runtime behavior* — they're excellent at catching implementation bugs like SQL injection or XSS. But they can't reason about **business logic or design intent** — e.g., they won't tell you that a password-reset flow lets you reset another user's password because the design itself forgot to bind the reset token to the requesting user's identity. That's a *design flaw*, invisible to a scanner that has no concept of "should this token belong to this user." Threat modeling catches that *before* it's even coded, which is both cheaper and addresses classes of bugs scanners are structurally blind to.

**Q22: How would you prioritize threats when you have limited time/resources to threat model an entire complex application?**
A: Focus on:
1. **Crown jewels first** — components handling the most sensitive data or critical business functions.
2. **New/changed components** — recently modified architecture is statistically more likely to have unaddressed threats.
3. **External-facing trust boundaries** before internal ones (internet-facing attack surface is the most reachable by external attackers).
4. **High blast-radius components** — e.g., a shared auth service used by 50 other services, where compromise has outsized impact.
Use a quick risk-scoring pass (even informal High/Med/Low) to triage before doing deep STRIDE analysis only on the top-tier items.

**Q23: What's the relationship between threat modeling and MITRE ATT&CK, and how would you use both together?**
A: Threat modeling (STRIDE/PASTA) is a design-time, top-down approach — it tells you *what categories of threats* are theoretically possible given your architecture. MITRE ATT&CK is a bottom-up, empirical catalog of *how real attackers have actually executed* those threat categories in observed incidents. In practice: after identifying via STRIDE that "Elevation of Privilege via the API Gateway" is a relevant threat, you'd consult ATT&CK techniques (e.g., T1078 "Valid Accounts," T1548 "Abuse Elevation Control Mechanism") to understand realistic attacker TTPs for that threat, which then informs both detection engineering (SOC) and more realistic red-team/pen-test scenarios.

**Q24: How do you handle disagreement between a developer and security team on whether a threat is "realistic enough" to fix?**
A: Bring it back to a structured risk score (likelihood × impact, or DREAD) rather than opinion — quantify discoverability (is it in the OWASP Top 10? Has it been seen in the wild via threat intel/CVE data?) and exploitability (does it require authenticated access, specific timing, etc.?). If genuine disagreement remains after that, escalate to a formal **risk acceptance** process with business-owner sign-off, so the decision and its owner are documented rather than silently dropped.

**Q25: Your organization wants to start a "continuous threat modeling" practice. What does that look like in practice, and what changes from traditional (point-in-time) threat modeling?**
A: Instead of a single big threat-modeling workshop at project kickoff, continuous threat modeling treats the model as a living artifact:
- A baseline threat model is created early, stored in version control (often as code via pytm/Threagile or in a wiki linked to the architecture decision records).
- Any PR/architecture change that touches a trust boundary, adds a new external integration, or introduces a new data store triggers a lightweight re-review (sometimes automated via CI checks looking for IaC/architecture diffs).
- Smaller, more frequent "threat modeling moments" (e.g., 30 minutes per sprint on the riskiest change) replace one large annual exercise.
- Findings are tracked as backlog items with the same rigor as bugs, not as a one-time PDF report that gets forgotten.

## 19. Glossary (Quick Reference)

| Term | One-line definition |
|---|---|
| AINCAA | Authentication, Integrity, Non-repudiation, Confidentiality, Availability, Authorization — the properties STRIDE threats violate |
| BOLA | Broken Object Level Authorization (API1) |
| BFLA | Broken Function Level Authorization (API5) |
| IDOR | Insecure Direct Object Reference — a common manifestation of BOLA |
| SSRF | Server-Side Request Forgery — server tricked into making unintended requests |
| Zero Trust | Security model assuming no implicit trust, even inside the network perimeter |
| Blast Radius | Scope of impact if a component is compromised |
| Crown Jewels | Highest-value assets in a system |
| Shift Left | Moving security activities earlier in the SDLC |
| CWE | Common Weakness Enumeration — catalog of weakness types (used to classify SAST findings) |
| CVE | Common Vulnerabilities and Exposures — catalog of specific known vulnerabilities (used in SCA) |
| CVSS | Common Vulnerability Scoring System — standardized severity scoring |
| TTP | Tactics, Techniques, and Procedures (used in MITRE ATT&CK) |
| mTLS | Mutual TLS — both client and server authenticate each other via certificates |
| RBAC / ABAC | Role-Based / Attribute-Based Access Control |
| SDL | Security Development Lifecycle (Microsoft's framework, origin of STRIDE) |
